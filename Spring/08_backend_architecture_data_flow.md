
#  백엔드 아키텍처와 데이터 흐름의 이해

> **Date:** 2026-03-12 <br>
> **Topic:** 의존성 주입(DI), DTO 설계, Java Stream API, Optional과 예외 처리, 테스트 코드


## 1. 의존성 주입(DI): 객체 간의 결합도를 낮추는 기술

단순히 `@Autowired`를 쓰는 것보다 **생성자 주입**을 권장하는 이유를 체득.

* **안전성:** `final` 키워드를 사용하여 객체가 생성된 후 변하지 않음을 보장.
* **테스트 용이성:** 가짜 객체(Mock)를 갈아 끼우기 쉬워 유닛 테스트에 유리.
* **Tip:** `@RequiredArgsConstructor`는 `final`이 붙은 필드만 골라서 생성자를 만들어주는 어노테이션.

## 2. DTO(Data Transfer Object) 설계
엔티티는 DB와 1:1로 매핑되는 **'원본 장부'**. <br>
엔티티를 클라이언트에게 그대로 보여주면 안 되는 이유.

* **보안:** `Member` 엔티티의 비밀번호나 주민번호가 API 응답에 섞여 나가는 보안 문제 방지.
* **순환 참조 방지:** `Member -> Chat -> Member...` 식으로 서로를 무한 호출하다 서버가 뻗는 현상을 DTO 변환 과정에서 끊어줌.
* **유연성:** DB 컬럼명이 `user_nickname`에서 `user_id`로 바뀌어도, DTO만 잘 관리하면 클라이언트에게 전달하는 데이터 형식은 유지 가능.

## 3. Java Stream API
자바 8부터 도입된 기능으로, 컬렉션(List, Set, Map 등)에 저장된 요소들을 하나씩 참조하며 반복적인 처리를 가능하게 해주는 **'데이터 흐름'**. `for`문이나 `if`문 위주의 명령형 프로그래밍에서 벗어나, **선언적**으로 데이터를 처리.


### 1. 핵심 3단계 공정

#### ① 생성: `stream()`
* **비유:** 창고(List)에 쌓여있는 원재료들을 컨베이어 벨트 위로 올리는 과정.
* **설명:** 컬렉션 데이터를 흐르는 강물(Stream) 형태로 만듦. 이제부터 각 데이터는 한 줄로 서서 가공 공정을 지나게 됨.

#### ② 가공(중간 연산): `map()`
* **비유:** 컨베이어 벨트를 지나가는 원재료를 깎고 다듬어 부품(DTO)으로 만드는 공정.
* **설명:** `map`은 스트림 내부의 요소를 하나하나 꺼내서 특정 형태로 변환. 
    * **핵심 활용:** 엔티티(Entity) 객체에서 필요한 데이터만 뽑아 DTO로 변환하거나, `c.getSender().getName()`처럼 연관된 객체의 필드 값을 추출.

#### ③ 수거(최종 연산): `collect(Collectors.toList())`
* **비유:** 가공이 완료된 부품들을 다시 완성품 박스(새 리스트)에 담는 과정.
* **설명:** 스트림 흐름을 종료하고 결과물을 다시 컬렉션 형태로 되돌림. 이 단계가 실행되어야만 앞선 `map` 공정들이 실제로 동작(지연 연산).


### 2. 장점

1.  **가독성 향상:** 수십 줄의 `for`문과 내부 `if` 조건문이 단 몇 줄의 체이닝(`.`으로 연결)으로 끝.
2.  **선언적 코드:** '어떻게' 반복할지가 아니라, 데이터를 '어떻게 변형할지'에만 집중.
3.  **유연한 연관관계 처리:** JPA 등을 사용할 때 엔티티 계층 구조를 따라가며 필요한 데이터만 쏙 뽑아내기 편리.

### 3. 코드 요약

```java
// AS-IS: 기존의 for문 방식 (길고 복잡함)
List<UserDto> dtoList = new ArrayList<>();
for (User user : users) {
    dtoList.add(new UserDto(user.getName()));
}

// TO-BE: Stream 방식 (간결하고 명확함)
List<UserDto> dtoList = users.stream()                // 1. 강물 만들기
    .map(user -> new UserDto(user.getName()))        // 2. DTO로 가공하기
    .collect(Collectors.toList());                   // 3. 바구니에 담기
```

## 4. Optional과 예외 처리

데이터가 없을 때 발생하는 `NullPointerException`.

* **`Optional<T>`**: 값이 있을 수도, 없을 수도 있다는 것을 코드로 명시.
* **`.orElseThrow()`**: "데이터가 없으면 비즈니스 로직을 더 진행하지 말고, 예외(Exception)를 발생시켜라"라고 명확하게 선언하여 코드의 안정성을 높임.


## 5. 테스트 코드: 단위 vs 통합

* **단위 테스트:** 메서드 하나가 수학적으로 완벽하게 돌아가는지 확인. (빛의 속도)
* **통합 테스트:** 실제 DB를 붙여서 "A가 B에게 메시지를 보냈을 때 DB에 잘 쌓이는지" 전체 흐름을 봄. (신뢰도 높음)
