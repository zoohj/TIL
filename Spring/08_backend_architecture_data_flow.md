
#  백엔드 아키텍처와 데이터 흐름의 이해

> **Date:** 2026-03-12
> **Topic:** 의존성 주입(DI), DTO 설계, Java Stream API, Optional과 예외 처리, 테스트 코드


## 1. 의존성 주입(DI): 객체 간의 결합도를 낮추는 기술

단순히 `@Autowired`를 쓰는 것보다 **생성자 주입**을 권장하는 이유를 체득했습니다.

* **안전성:** `final` 키워드를 사용하여 객체가 생성된 후 변하지 않음을 보장합니다.
* **테스트 용이성:** 가짜 객체(Mock)를 갈아 끼우기 쉬워 유닛 테스트에 유리합니다.
* **Tip:** `@RequiredArgsConstructor`는 `final`이 붙은 필드만 골라서 생성자를 만들어주는 롬복의 효자 어노테이션입니다.

## 2. DTO(Data Transfer Object) 설계: 왜 엔티티를 꽁꽁 숨겨야 할까?

엔티티는 DB와 1:1로 매핑되는 **'원본 장부'**입니다. 이걸 손님(클라이언트)에게 그대로 보여주면 안 되는 이유를 정리했습니다.

* **보안:** `Member` 엔티티의 비밀번호나 주민번호가 API 응답에 섞여 나가는 대참사를 방지합니다.
* **순환 참조 방지:** `Member -> Chat -> Member...` 식으로 서로를 무한 호출하다 서버가 뻗는 현상을 DTO 변환 과정에서 끊어줍니다.
* **유연성:** DB 컬럼명이 `user_nickname`에서 `user_id`로 바뀌어도, DTO만 잘 관리하면 클라이언트에게 전달하는 데이터 형식은 유지할 수 있습니다.

## 3. Java Stream API: 노가다 반복문에서 탈출하기

리스트 안에 든 수많은 데이터를 하나하나 꺼내서 DTO로 옮겨 담는 과정을 단 몇 줄로 줄였습니다.

* **`stream()`**: 리스트를 흐르는 데이터의 강물로 만듭니다.
* **`map()`**: 강물을 지나가는 엔티티 하나하나를 `new DTO()`로 가공합니다.
* **`collect(Collectors.toList())`**: 가공된 DTO들을 다시 예쁜 리스트 바구니에 담습니다.
* **핵심:** 이 과정에서 `c.getSender().getName()`처럼 연관관계를 활용해 필요한 데이터만 쏙 뽑아내는 센스가 중요합니다.

## 4. Optional과 예외 처리: "없으면 에러 던져!"

데이터가 없을 때 발생하는 `NullPointerException`은 백엔드 개발자의 숙적입니다.

* **`Optional<T>`**: 값이 있을 수도, 없을 수도 있다는 것을 코드로 명시합니다.
* **`.orElseThrow()`**: "데이터가 없으면 비즈니스 로직을 더 진행하지 말고, 예외(Exception)를 발생시켜라"라고 명확하게 선언하여 코드의 안정성을 높입니다.

## 5. 테스트 코드의 철학: 단위 vs 통합

* **단위 테스트:** 메서드 하나가 수학적으로 완벽하게 돌아가는지 확인합니다. (빛의 속도)
* **통합 테스트:** 실제 DB를 붙여서 "A가 B에게 메시지를 보냈을 때 DB에 잘 쌓이는지" 전체 흐름을 봅니다. (신뢰도 높음)
