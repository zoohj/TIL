# AOP를 활용한 공통 관심 사항 분리

> **Date:** 2026-03-09 <br>
> **Topic:** AOP가 필요한 상황, 적용 

---


## 1. AOP가 필요한 상황

모든 메서드의 호출 시간을 측정해야 하는 상황을 가정.

* **핵심 관심 사항(Core Concern)**: 회원 가입, 회원 조회 등 실제 비즈니스 로직.


* **공통 관심 사항(Cross-cutting Concern)**: 시간 측정 로직처럼 여러 메서드에 공통으로 적용되는 부가 기능.


* **문제점**: 시간 측정 로직을 모든 메서드에 직접 넣으면 비즈니스 로직과 섞여 유지보수가 매우 어렵고, 로직 변경 시 모든 코드를 다 수정해야 함.



## 2. AOP(Aspect Oriented Programming)란?

공통 관심 사항과 핵심 관심 사항을 분리하여 원하는 곳에 공통 기능을 적용하는 기술.

## 3. AOP 적용 방법

`@Aspect` 애노테이션을 사용하여 공통 로직을 별도의 클래스로 만듦.

## 4. AOP의 동작 원리 (프록시 기술)

스프링은 AOP가 적용되면 실제 객체 대신 **가짜 객체(프록시, Proxy)**를 생성하여 스프링 빈으로 등록.

* **호출 흐름**: 컨트롤러가 서비스를 호출할 때, 실제 서비스가 아닌 프록시 서비스를 먼저 호출.


* **실행 과정**: 프록시 서비스에서 공통 로직(시간 측정 등)을 실행한 후, `joinPoint.proceed()`가 호출되면 그제야 실제 서비스의 로직이 실행.


* **확인 방법**: 의존관계 주입 시 실제 클래스가 아닌 `$$EnhancerBySpringCGLIB` 같은 이름의 프록시 객체가 주입되는 것을 콘솔에서 확인할 수 있음.

## 💡 AOP의 장점
> * **비즈니스 로직을 깔끔하게 유지하면서 공통 기능을 효율적으로 관리할 수 있음.** 
> * 공통 로직을 변경하고 싶을 때 AOP 클래스 한 곳만 수정하면 전체에 반영.
