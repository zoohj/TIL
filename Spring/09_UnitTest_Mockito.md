# 단위테스트

> **Date:** 2026-03-18 <br>
> **Topic:** 단위 테스트(Unit Test), Mockito


## 1. 단위 테스트의 정의 (What)
* **개념:** 소스 코드의 가장 작은 단위(주로 메서드 하나)를 고립시켜서 의도대로 작동하는지 확인하는 테스트.

## 2. 왜 단위 테스트를 하나요? (Why)
1.  **속도:** 스프링(톰캣)을 띄우지 않고 자바 로직만 실행.
2.  **정확한 버그 위치:** 통합 테스트는 에러가 나면 "어디가 고장 났지?" 찾아야 하지만, 단위 테스트는 실패한 그 지점이 바로 범인입니다.
3.  **불안감 해소:** 코드를 수정(리팩토링)한 후 테스트를 돌려 "초록불"이 뜨면, 내 수정이 기존 기능을 망가뜨리지 않았음을 확신할 수 있습니다.


## 3. Mockito: 가짜 객체 활용법
단위 테스트는 '고립'이 생명입니다. `ChatRoomService`를 테스트할 때 진짜 DB나 다른 서비스를 쓰면 안 되기 때문에 **가짜 배우(Mock)**를 세웁니다.



* **`@Mock`:** 가짜 객체를 만듭니다. (예: `MemberRepository`)
* **`@InjectMocks`:** 가짜 객체들을 주입받을 주인공입니다. (예: `ChatRoomService`)
* **`when(...).thenReturn(...)`:** 가짜 객체에게 "누가 1번 찾으면 이 객체를 줘!"라고 **대본**을 짜주는 것입니다.
* **`verify(...)`:** "너 진짜로 저장(save) 버튼 눌렀어?"라고 **행동을 감시**하는 것입니다.


## 4. 테스트 작성 공식: Given - When - Then
가장 표준적인 작성 흐름입니다.



1.  **Given (준비):** 테스트에 필요한 데이터와 가짜 객체의 대본(`when`)을 준비합니다.
2.  **When (실행):** 실제 테스트하고 싶은 메서드(서비스 로직)를 호출합니다.
3.  **Then (검증):** 결과값이 맞는지(`assertThat`), 특정 메서드가 실행됐는지(`verify`) 확인합니다.



## 5. 서비스 코드에서 테스트 포인트를 찾는 법 (현지님의 눈!)
서비스 코드를 볼 때 다음 세 군데에 형광펜을 칠한다고 생각하세요.

* **외부 호출:** `repository.save()` 등이 있는가? -> **`verify` 대상**
* **분기점:** `if`나 `orElseThrow`가 있는가? -> **테스트 케이스(성공/실패)를 나눌 지점**
* **반환값:** `return` 하는 값이 있는가? -> **`assertThat`으로 값 검증 대상**


## 6. 트러블슈팅
* **`UnnecessaryStubbingException`:** 대본(`when`)은 짜놨는데 실제 코드에서 그 대본을 안 읽었을 때 발생합니다. (안 쓰는 대본은 지워야 깔끔해요!)
* **`protected access`:** 엔티티 생성자가 막혀있을 땐 `@Builder`를 사용해 객체를 만드는 것이 정석입니다.
* **`any(Class)` 오타:** `verify` 할 때는 '저장소'가 아니라 저장되는 '엔티티' 클래스를 적어야 합니다.
