# JPA 연관관계 매핑과 데이터 공정 설계

> **Date:** 2026-03-11
> **Topic:** 도메인 모델 설계, 연관관계 매핑($N:1$), 데이터 타입 변환(DTO to Entity)


## 1. 연관관계의 주인과 물리 설계 (`$N:1$`)

* **핵심 개념**: 외래 키(FK)를 가진 `Task` 엔티티가 연관관계의 주인이 됨.
* **설계 전략**: Java 필드명은 `member`로 객체답게 참조하고, DB 컬럼명은 `@JoinColumn(name = "member_id")`로 명시하여 관례를 따름.
* **성능 최적화**: `@ManyToOne(fetch = FetchType.LAZY)`를 적용하여, 불필요한 시점에 Member 정보까지 불러오는 'N+1 문제'를 사전에 방지함.

## 2. 다대다($N:M$) 관계의 해법: 매개 엔티티 (`ChatRoomMember`)

* **문제점**: 객체는 양방향 리스트로 $N:M$ 표현이 가능하지만, 관계형 DB는 두 테이블만으로 $N:M$을 저장할 수 없음.
* **해결책**: `Member`와 `ChatRoom` 사이에 **`ChatRoomMember`라는 연결 테이블**을 엔티티로 승격시켜 $1:N$, $N:1$ 관계로 풀어냄.
* **이점**: 단순히 '누가 어디에 있는지'를 넘어, '입장 시간', '권한(방장 여부)' 등 추가 메타데이터를 담을 수 있는 확장성 확보.

## 3. 데이터 타입 변환 공정 (`LocalDate` → `LocalDateTime`)

* **사용자 경험(UX)**: 사용자는 시/분/초 입력 없이 날짜(`2026-03-11`)만 입력하는 것이 편리함.
* **시스템 무결성**: 서버 내부와 DB에서는 정밀한 시간 계산 및 정렬을 위해 `LocalDateTime` 표준을 준수해야 함.
* **통역사 역할**: 서비스 레이어에서 `.atStartOfDay()`를 사용하여 입력값의 부족한 정보를 채워주는 '데이터 규격화'를 실현.

## 4. 테스트 코드: 비즈니스 로직의 '안전벨트'

* **가치**: 단순 저장 확인을 넘어, **[입력(DTO) → 변환(Service) → 영속화(Repository)]**로 이어지는 전체 공정 라인의 결함 여부를 체크.
* **회귀 테스트**: 향후 기능 고도화 시, 기존 로직이 망가지지 않았음을 단 한 번의 클릭으로 증명하는 유지보수의 핵심 자산.
